Every HTML page starts with a docutype declaration. Docutype's purpose is to tell the browser what version of HTML to render the document in. As of 3/18/24 the latest version is HTML5.

After we declare the docutype, we need to provide an html element. This is what;s known as the root element of the document, meaning every other element in the doument will be a descendent of it. 'lang' is an HTML attribute associated with the tag html. lang="en" is used to signify english and allows assistive technologies like screen readers to adapt accordingly to language and correct pronunciation.

Head Element

'head' Used for important meta-information about the webpages, and stuff required for our webpages to render correctly. You SHOULD NOT use any element that displays content on the webpage within the head element.

Meta Element

' meta' We should always have the meta tag with the charset encoding of the webpage in the head element. charset="UTF-8" is the default for most languages. This encoding setting is very important because it ensures the webpage will display special symbols and characters from different languages correctly in the browser.

Title Element

'title' Should also be in every head of a HTML document. This element is used to give the webpage a human readable Title to be used in the webpage's browser tab.

Body Element

'body' is the final element in the HTML boilerplate. This is where all the content that will be displayed to users will go: text, images, lists, links, and so on.

Links and Images

Anchor Elements: To create a link in HTML, we use the anchor element. An anchor element is defined by wrapping the tesct or another HTML element iwe want to be a link with an "a" tag.

  • Clicking on the newly created link does nothing. In order for the link to work we have to tell it a destination to go to by using an HTML attribute
  • An HTML attribute gives the additional information to an HTML element and always goes in the element's opening tag. An attribute is usually made up of 2 parts
    1. A name
    2. A value
  • In this case we need to add an href(hypertext reference) attribute to the opening anchor tag. This href attribute is the destination we would like to link to.
  • By default any text wrapped with and anchor tag not containing the href attribute will look like plain text. With the href tag present the browser will give the text a blue color and underline it to signify it is a link.
  • You can use anchor tages to link to any kind of resource on the internet, not just other HTML documents. You can link videos, pdf files, images, and so on.
  • By default the link will open in the current open tab. If you want the link to open in a place other than the current tab you must add a target attribute to the href element
    • The default to open in the same tab target="_self"
    • To open in a new tab or window target="_blank"
  • The rel attribute is used to describe the relation between the current page and the linked document
    • rel="noopener" prevents the opened link from gaining access to the webpage from whish it was opened.
    • rel="noreferrer" prevents the opened link from knowing which webpage or resource has a link (or 'reference') to it. It also includes the noopener behaviour and thus can be used by itself as well.
  • These two rel attributes are in place for additional security. The prevention of access created with the "noopener" prevents phishing attacks wher the opened link may change the original webpage to a different one to trick users. This is refered to as tabnabbing. Adding the "noreferrer" value can be done if you wish to not let the opened link know that your webpage links to it.
  • More recent versions of browsers have security built in to prevent these types of attacks if you do not include the rel attributes, but it is always recommended to pair a 'target="_blank"' with the two rel attributes.

Absolute and Relative Links

Absolute Links: Links to pages on other websites on the internet are called absolute links. A typical absolute link will be made up of the following parts.

  • protocal://domain/path
  • Ex: https://www.theodinproject.com/about
Relative Links: These are links to other pages within our own website. These links do not include the domain name as it assumes the domain name will be the same since it is a page on the same site. These links only include the file path to the other page.

Images

To display an image in HTML we use the img element. The img element is self closing, meaning that you do not need to include a closing tag. Instead of wrapping the content, it embeds an image into the page using a src attribute that tells the browser where the image file is located. This src attribute works much like the href attribute for anchor tags and embeds the image with the use of either absolute or relative paths.

  • Parent Directories: What if we wanted to use the image in a page that is in a outside of the current directory? To get back to the parent directory we us (Two) .. in the filepath instead of just (one) .
    • Ex: 'img src="../images/Unicorn Rampage Logo.jpg"
  • Alt Attribute: Besides the src attribute, every image element must also have an alt (alternative text) attribute.
    • The alt attribute is used to describe an image
    • It is used in place of the image if the image cannot be loaded
    • It is also used with screen readers to describe what the image is to visually impaired users
  • Height and Width Tags: You can specify the height and width of an image in the attributes. This helps the browser layout the page without causing the page to jump and flash. It is good practice to include these attributes even when the image is already the correct size or you are using CSS to modify it

There are 4 main image formats that are in use on the web. Each was designed to do different things.

  1. JPG Images: These images are designed for handling large color palettes without dramatically increasing size of the file. This attribute makes them great for photos and images with lots of color gradients in them. They do not allow for transparent pixels though.
  2. GIF Images: These are the go-to for simple animations. They are limited when it comes to color palette. They should never be used for photos. Transparent pixels are a binary option for GIFs, meaning you can't have semi-opaque pixels. This can make it difficult to get high levels of detail ona transparent background.
  3. PNG Images: These images are great for anything that is not a photo or animated. They deal with opacity just fine and do not have color palette limitations making them excellent for use in icons, technical diagrams, & logos
  4. SVG Images: These images are not pixel based as the other three types are. They are Vector-based graphics. This means they can scale up or down to any dimension without loss of quality. This attribute makes them great for Respnsive design. They can be used in almost all the same cases as PNGs and should be used whenever possible. The downside to their use is for them to display consistently across browsers, you must convert any text fields to outlines using your image editor. If your image contains a lot of text this can have a huge impact on file size.

Commit Messages

What is important about Commit Messages?

  1. When you are applying for jobs, employers will look through your projects on GitHub and they will look through your commit history. Having good commits as a novice developer will help you stand out.
  2. Having good commit message history will allow you (or other developers working on your code) to quickly see what changes were made and why. This is useful if a bug is found in the code that needs to be fixed!
  3. Having a good commit message history will also be helpful if you come back to a project you were working on after stepping away from it for a while. You likely won't remember your thought process and changes made when initially writing out the code.
Solid effective commits will contain two separate parts. A Subject and a Body
  • Subject: A brief summary of the change you made to the project.
    • Ex: This is the change I made to the codebase.
  • Body: A concise yet clear description of what you did.
    • Ex: Describe the problem your commit solves and how.
  • Full Example of a Good Commit Message
    1. | Add missing link and alt text to the company's logo
    2. |
    3. |Screen readers won't read the images to users with disabilities without this information

    This message:

    • Provides a subject that specifies your code's action
    • Contains a body that provides a concise yet clear description of why the commit needed to be made.
    • Separates the jusbject from the body with a new/blank line. This is best practice and should be the format used going forward.

When to Commit

  1. Everytime you have a meaningful change in the code. Creating a timeline of your progress and show that your finished code didn't appear out of nowwhere.
    • Get a piece of code to function as you like.
    • Fix a typo
    • Fix a bug

There will come a time when the project is FINALLY working as you want it to. This is when you should commit. Then later if it breaks and you don't know what changed to break it you can revert back to that commit when it all worked as you intended and see what it looked like then and what changed since.
